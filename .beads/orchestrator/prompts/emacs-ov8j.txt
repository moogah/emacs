
○ emacs-ov8j · Optimize list operations in bash-parser recursive analyzer   [● P1 · OPEN]
Owner: Jeff Farr · Type: task
Created: 2026-03-06 · Updated: 2026-03-06

DESCRIPTION
Fix excessive list copying in hot paths of bash-parser recursive analyzer.

Current implementation uses copy-tree and copy-sequence repeatedly when marking operations with metadata, causing O(n×m) complexity where n=operations, m=average operation size.

Files to modify:
- config/experiments/bash-parser/bash-parser-recursive.el (primary)
- config/experiments/bash-parser/bash-parser-file-ops.el (secondary)

Problem locations:

1. jf/bash-analyze-file-operations-recursive (line 154-158)
   Current pattern:
   ```elisp
   (dolist (op subst-ops)
     (let ((marked-op (copy-tree op)))
       (plist-put marked-op :from-substitution t)
       (push marked-op marked-ops)))
   ```
   
2. jf/bash--extract-conditional-context-operations (lines 494, 528, 560)
   Same copy-tree pattern repeated for test/then/else branches

3. Pattern appears ~15 times across recursive analyzer

Implementation fix:
Replace copy-tree pattern with direct plist construction:

```elisp
;; BEFORE (O(n×m) - creates full copy)
(let ((marked-op (copy-tree op)))
  (plist-put marked-op :from-substitution t)
  (push marked-op operations))

;; AFTER (O(n) - builds correctly first time)
(push (append op (list :from-substitution t)) operations)
```

Alternative for complex metadata:
```elisp
;; When adding multiple fields
(push (append op (list :from-substitution t
                      :loop-context t
                      :loop-variable loop-var))
      operations)
```

Locations to fix:
1. Line 154-158: Substitution marking (copy-tree → append)
2. Line 300-302: Loop context marking (copy-tree → append)
3. Line 324-326: Subshell marking (copy-tree → append)
4. Line 494: Test condition marking (copy-tree → append)
5. Line 528: Then branch marking (copy-tree → append)
6. Line 560: Else branch marking (copy-tree → append)

Design rationale:
- Operations are created once and never mutated after
- Plists are immutable once constructed
- append creates new list without copying nested structures
- Eliminates O(m) copy cost per operation
- Hot path in recursive analyzer (called many times)

Performance impact:
- Before: O(n×m) where m can be 10-20 fields per operation
- After: O(n) - only cons cells allocated
- Expected: 5-10x speedup for commands with many operations
