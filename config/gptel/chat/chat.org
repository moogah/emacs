* Introduction

This file is the loader for the =gptel-chat= subsystem. The chat-mode
subsystem introduces a dedicated major mode (=gptel-chat-mode=) for
conversational LLM buffers, replacing the ad-hoc =gptel-mode= usage in
persistent sessions.

The subsystem is factored into cohesive feature modules, each with a
narrow responsibility and small public surface (see
=openspec/changes/gptel-chat-mode/architecture.md= §Components):

- =mode=     - Major mode, keymap, font-lock, activation hooks
- =parser=   - Buffer walker producing a turn list (user/assistant)
- =stream=   - Streaming callback and accumulator state machine
- =send=     - Interactive =gptel-chat-send= and =gptel-request= wiring
- =nav=      - Turn navigation and regenerate-last-response
- =display=  - Presentation layer (faces, overlays, toggle)
- =menu=     - Transient prefix + preset-application wiring

This loader only loads those feature modules. Their actual
implementation lives in the sibling =.org= files and is delivered by
later tasks in the =gptel-chat-mode= OpenSpec change — at scaffold
time, each module is a stub that only calls =provide=.

* Lexical Binding

Keep =lexical-binding: t= on line 1 of =chat.el= — Emacs only honors
the mode line when it is the very first line of the file (see the
project memory on lexical-binding regressions). The =:comments no=
header tells org-babel not to emit structural comments above this
block during tangling.

* Feature Module Loading

Load the chat-mode feature modules in dependency order. The loader
follows the same pattern as =config/gptel/gptel.org= — serial
=jf/load-module= calls against absolute paths resolved from
=jf/emacs-dir=.

Dependency notes (from architecture.md §Interfaces):

1. =mode= declares the major mode and keymap — every other module
   assumes it exists.
2. =parser= walks the buffer; it depends only on =mode=.
3. =stream= builds the streaming callback; its internal
   =gptel-chat--sanitize-chunk= handles delimiter-collision escaping
   (see architecture.md §Components), and it uses =parser=.
4. =send= wires =parser= and =stream= to =gptel-request=.
5. =nav= loads after =send= because =regenerate= is a thin wrapper
   over =gptel-chat-send= — it deletes the trailing assistant block
   and re-invokes the send entry point. Turn navigation itself uses
   =re-search-forward= on block delimiters and does NOT depend on
   =parser= internals (architecture.md §gptel-chat-nav explicitly
   denies coupling to parser internals).
6. =display= is presentation-only; safe to load last among the core
   modules.
7. =menu= defines the transient prefix and the preset-application
   hook; it depends on =send= (for the rebound Send suffix).

(jf/load-module (expand-file-name "config/gptel/chat/mode.el" jf/emacs-dir))
(jf/load-module (expand-file-name "config/gptel/chat/parser.el" jf/emacs-dir))
(jf/load-module (expand-file-name "config/gptel/chat/stream.el" jf/emacs-dir))
