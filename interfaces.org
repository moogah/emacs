* About this register

This file is the authoritative catalogue of the contracts that flow across
module boundaries in the isolated emacs config. It is the artifact the
opsx-orchestrator's Architect role audits against in three triggers:

- *On-touch* (in execute) — when an Implementor diff modifies code cited
  in a =load_bearing: true= entry, the Architect runs against that entry
  only.
- *End-of-cycle* (in integrate) — full signal-class run across the cycle's
  diffs and the register.
- *Between-cycle* (=/architect-audit=, on demand) — register vs. whole repo;
  consolidation-round-as-a-button.

Each entry is a YAML-fenced block under one of the four tier-headings below,
following the templates at:

- =~/.claude/skills/opsx-orchestrate/templates/register-entry-shape.md=
- =~/.claude/skills/opsx-orchestrate/templates/register-entry-vocabulary.md=
- =~/.claude/skills/opsx-orchestrate/templates/register-entry-boundary.md=
- =~/.claude/skills/opsx-orchestrate/templates/register-entry-invariant.md=

Per-entry =status= is one of =speculated | confirmed | divergent |
reconciled=. =load_bearing: true= signals the entry should receive on-touch
Architect attention whenever cited code changes. New entries land as
=speculated= during plan-phase forward-mode and are reconciled at integrate
based on Implementor =## Discoveries=.

Lifecycle reference: =~/.claude/skills/opsx-orchestrate/SKILL.md=,
=scaffolding.md=, =flows/{plan,execute,integrate}.md=.

* Conventions

- *One entry per concept.* If two entries describe the same shape with
  different field sets, that's drift — reconcile, don't duplicate.
- *Tier discipline.* Before creating an invariant entry, check whether the
  rule fits =shape=, =vocabulary=, or =boundary= — those tiers carry their
  own enforcement; the invariant tier is the safety net for rules that
  don't fit structurally.
- *Cite specific files and functions.* "the validator" is not a citation;
  =config/gptel/scope/scope-validation.org=, function
  =jf/gptel-scope--load-config= is.
- *Past incidents anchor entries.* The =why_it_matters= field of an
  invariant is most credible when it names an incident this project
  has actually paid for.

* Shape

  Entries describe plists, alists, structs, records that cross module
  boundaries. Producers and consumers are listed; the canonical
  constructor (if any) is the single allowed producer.

     prepend to the chat-mode initial content when writing a fresh
     session.org. Producer is jf/gptel-scope-profile--render-drawer-text;
     consumers are session-creation (sessions/commands.org) and persistent-
     agent creation (tools/persistent-agent.org). The shape is "an org

     - anchor: ":PROPERTIES:"
       must_appear: at byte 0 (or after a leading whitespace-only prefix)
     - anchor: ":END:"

     - anchor: ":GPTEL_PRESET:"

     - every line between :PROPERTIES: and :END: matches the regex

     - the trailing newline after :END: is included (so callers can simply
       concat with the chat-mode initial content)

   producers:
     - file: config/gptel/scope-profiles.org

       function: <session-creation entrypoint>
       usage: same
     - file: config/gptel/tools/persistent-agent.org
       function: <agent-creation function around persistent-agent.el:111>

             ((not (string-match-p "\\`:PROPERTIES:\n" val))
              'missing-properties-header)
             ((not (string-match-p "\n:END:\n\\'" val))
              'missing-end-or-trailing-newline)
             ((/= 1 (cl-count-if (lambda (line)
                                   (string= line ":PROPERTIES:"))
                                 (split-string val "\n")))
              'duplicated-properties-header)
             (t nil)))

* Vocabulary

  Entries describe enumerations / symbol sets / value sets that span a
  boundary, plus the canonical mapping (=pcase= / dispatch table) between
  layers. The motivating failure is bash-parser semantic-op set ↔
  scope section keys: a =pcase= inlined at multiple sites, missing values
  at one of them, silently mis-routing operations.

   purpose: |

     jf/gptel-scope--map-operation-to-drawer-key — and any pcase / cond

     GPTEL_SCOPE_DENY is *not* a collapse target: deny-listing is
     expressed at the action layer (an explicit "add to deny scope"
     handler), not as an :operation input. A :deny arm in this mapping

     conflate violation-outcome (denied) with operation-kind (read/write/

     bash-parser emits :match-pattern with the literal pattern as
     :resource (e.g. "*.txt" from `find -name`); routing the pattern
     into a path-allowlist drawer key would grant reads on every
     filesystem match. Cycle-4 (Ask 1) resolved this as Option B:
     the action handler refuses with a directing user-error
     ("use c for custom pattern, e for manual edit") rather than
     redirecting. The cycle-2 ask-10B "smart redirect" is not
     implementable in cycle-3-as-shipped because
     `--validate-file-operations` (scope-validation.el:412) throws on
     the first denial via `(catch 'error-found ...)`, so the action
     handler never receives a full violation cluster from which to
     pick a sibling :read-directory resource. The writer's strict-error
     fallback therefore signals "match-pattern reached the writer" as
     a defect, not a vocabulary gap.
