* About this register

This file is the authoritative catalogue of the contracts that flow across
module boundaries in the isolated emacs config. It is the artifact the
opsx-orchestrator's Architect role audits against in three triggers:

- *On-touch* (in execute) — when an Implementor diff modifies code cited
  in a =load_bearing: true= entry, the Architect runs against that entry
  only.
- *End-of-cycle* (in integrate) — full signal-class run across the cycle's
  diffs and the register.
- *Between-cycle* (=/architect-audit=, on demand) — register vs. whole repo;
  consolidation-round-as-a-button.

Each entry is a YAML-fenced block under one of the four tier-headings below,
following the templates at:

- =~/.claude/skills/opsx-orchestrate/templates/register-entry-shape.md=
- =~/.claude/skills/opsx-orchestrate/templates/register-entry-vocabulary.md=
- =~/.claude/skills/opsx-orchestrate/templates/register-entry-boundary.md=
- =~/.claude/skills/opsx-orchestrate/templates/register-entry-invariant.md=

Per-entry =status= is one of =speculated | confirmed | divergent |
reconciled=. =load_bearing: true= signals the entry should receive on-touch
Architect attention whenever cited code changes. New entries land as
=speculated= during plan-phase forward-mode and are reconciled at integrate
based on Implementor =## Discoveries=.

Lifecycle reference: =~/.claude/skills/opsx-orchestrate/SKILL.md=,
=scaffolding.md=, =flows/{plan,execute,integrate}.md=.

* Conventions

- *One entry per concept.* If two entries describe the same shape with
  different field sets, that's drift — reconcile, don't duplicate.
- *Tier discipline.* Before creating an invariant entry, check whether the
  rule fits =shape=, =vocabulary=, or =boundary= — those tiers carry their
  own enforcement; the invariant tier is the safety net for rules that
  don't fit structurally.
- *Cite specific files and functions.* "the validator" is not a citation;
  =config/gptel/scope/scope-validation.org=, function
  =jf/gptel-scope--load-config= is.
- *Past incidents anchor entries.* The =why_it_matters= field of an
  invariant is most credible when it names an incident this project
  has actually paid for.

* Shape

  Entries describe plists, alists, structs, records that cross module
  boundaries. Producers and consumers are listed; the canonical
  constructor (if any) is the single allowed producer.

   statement: |

     content off column 0 at once, so org's column-0-anchored scanners

     regex parser is structurally indifferent — message construction
     still works — but the host buffer would otherwise become unusable

   tier: invariant
   status: speculated
   load_bearing: true
   title: body indent/dedent round-trip is content-preserving

   statement: |
     For any message segment destined for a chat-block body: applying
     the write-path indent (boundary stage 1) and then the parser dedent
     (boundary stage 2) yields content equal to the original modulo a
     uniform leading-indentation shift. The dedent measures and strips
     each segment's common minimum indentation, so it is robust against
     changes to `gptel-chat-content-indentation` and legacy content with
     different indent widths, and it preserves the relative indentation
     of nested content (lists, code) within the segment.

   why_it_matters: |

     what the user composes and what the LLM receives must be identical
     except for the on-disk body indent. A dedent that stripped a fixed
     assumed width would corrupt content under any width change; a dedent
     that stripped all leading whitespace would flatten intentional
     nested indentation. Measure-and-strip of the common minimum is the
     `org-do-remove-indentation` model and is the only form that
     round-trips both robustly and structure-preservingly. Known
     accepted edge case: a segment whose every line is indented loses
     its outermost indent level — identical to org src-block behaviour.

     at column 0 for the same parser reason (Decision 5 — Path C);

   title: The :broken plist tag is runtime-only — never serialized to disk

   statement: |
     For every workspace =ws= written to the persistence file:
     =(plist-member (workspace--serialize ws) :broken)= MUST return
     nil. The =:broken= tag SHALL be set only at load time (by the
     deserializer's =(file-directory-p :home)= check) or by the
     explicit =workspace--mark-broken= helper, and SHALL be cleared
     by =workspace--clear-broken= or by =workspace-re-anchor=. The
     tag is an OBSERVATION about the current filesystem state, not a
     persistent attribute of the workspace.

     Equivalently: the on-disk representation of a workspace is the
     same whether the workspace was marked broken at load time or
     not. Reading a workspace as broken, serializing without change,
     reading back: produces a non-broken workspace IF the directory
     has reappeared, broken IF still missing.

   why_it_matters: |
     If =:broken= were serialized, the system would need a state
     machine for clearing it — when does a workspace become "no
     longer broken"? At the next successful activation? At the next
     restart? On =workspace-re-anchor=? Each choice introduces a
     drift point. Keeping the tag ephemeral means the answer is
     trivial: "the next load checks =file-directory-p= afresh".

     The persistence-write path is also the only path that turns
     in-memory state durable; if it omits =:broken= by construction,
     no future code path can accidentally persist the tag (e.g. by
     marking a workspace broken in memory and then immediately
     calling =workspace-save=). The invariant is therefore a
     load-bearing property of the serializer's output, not just of
     the runtime model.

     Pairs with =register/vocabulary/workspace-broken-disposition=:
     the vocabulary defines what commands DO when invoked on a
     broken workspace; this invariant defines how the broken state
     LIVES — at runtime only. The combination makes broken-state
     handling auditable: every code path that reacts to broken is
     in the vocabulary; every code path that produces broken state
     is the deserializer's =file-directory-p= check or an explicit
     =mark-broken= call.
