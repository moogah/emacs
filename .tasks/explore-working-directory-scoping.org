* Part 1 — Current-state findings (grounded)

** The three-role model (the core lens)

A "directory" plays three distinct roles in a gptel session. Today only
one is named explicitly; a second is conflated into it by accident; the
third is computed and discarded.

#+begin_example
  Role 1  SESSION / METADATA ROOT   bookkeeping dir (session.org, system-prompt, drawer)
          - tracked explicitly as buffer-local jf/gptel--branch-dir
          - scope drawer is LOADED from here
  Role 2  WORK ROOT                  what cwd / relative paths SHOULD mean (the project)
          - should be `default-directory`; should be what the model is TOLD
          - TODAY: unset -> falls through to Role 1 (the bug)
  Role 3  SCOPE BOUNDARY             GPTEL_SCOPE_READ/WRITE/DENY globs
          - already cleanly separate, self-describing in the drawer
#+end_example

The whole bug class reduces to: *Role 2 is never set, so `default-directory`
silently falls through to Role 1.* Both chat sessions and agents open
their buffer with =find-file-noselect= on a file under the session tree,
and nobody sets =default-directory= afterward.

** The cwd bug's true nature: wrong path universe, NOT fail-open

Originally framed (in the .tasks item) as "scope is bypassed / ineffective."
Closer reading shows the opposite mechanism:

- With =default-directory= = the sandbox/bookkeeping dir, a *relative*
  path the model emits resolves *inside that dir*, not the project.
- The validator then judges a path that has nothing to do with what the
  model meant. The scope decision becomes *meaningless* — sometimes
  spuriously permissive, sometimes spuriously restrictive — never "about
  the right file."

#+begin_example
  model intends:   read project file  config/gptel/presets/minimal.md
  resolved vs cwd= sandbox:  ~/.gptel/sessions/.../<agent>/config/gptel/presets/minimal.md
  validator judges THAT path  ->  decision is about a file the model never meant
#+end_example

Implication: fixing =default-directory= is not a convenience — it is what
makes "the path the validator judges" equal "the path the model meant."
Only then is scope validation testable or meaningful. The
file_not_found-vs-scope_violation ordering question is downstream of this.

** What workspaces changes (the target paradigm)

=config/workspaces/= (newly merged) makes the *work root a first-class,
on-disk concept*. A workspace = a named context, 1:1 with a tab, anchored
to an absolute =:home= directory. gptel sessions now live *inside* it.

User-confirmed topology: *worktrees are colocated in the workspace dir as
siblings; the workspace =:home= IS the project dir.*

#+begin_example
  ~/emacs-workspaces/<name>/                  :home  ==  WORK ROOT (Role 2) == project dir
  ├── home.org                                dashboard; editable; #+TITLE overrides display name
  ├── .git/
  ├── sessions/                               gptel sessions live HERE (Role 1, inside :home)
  │   └── <id>/branches/main/session.org
  │       └── agents/<agent>/session.org      agent sandbox, also inside :home
  ├── <worktree-feature-a>/  (.git is a FILE) ┐ sibling worktrees, enumerable
  └── <worktree-feature-b>/  (.git is a FILE) ┘
#+end_example

Key consequences:
- =${project_root}= now expands to =:home= (=workspace-integration.org:85=
  passes =project-root=home= into =create-session-core=). So *Role 3 (scope)
  is correct by construction in this path*: =WRITE: <:home>/**= covers
  sessions, worktrees, everything. The part feared broken isn't, in this flow.
- Of the three roles, workspaces solves two (metadata under :home, scope
  rooted at :home). *Only Role 2 (work root / default-directory) is still
  on the floor.*

Workspaces seam facts (one-way, decoupled):
- gptel consults =workspace-sessions-dir= behind =(featurep 'workspaces)= +
  =fboundp= guards (=filesystem.org:76-91=); =jf/gptel-persistent-session-global=
  forces global =~/.gptel/sessions/=.
- Integration registry: =workspace-register-integration= with =:on-create= /
  =:menu= / =:on-purge=; anchor payload plist =(:name :home :sessions-dir :context)=
  (=integrations.org=). gptel registers =gptel-session= in
  =workspace-integration.org= under =with-eval-after-load 'workspaces=.
- Workspaces ALREADY enumerates sibling worktrees:
  =jf/workspace--worktree-children= (=git-worktrees.org:235=) returns
  =(:dir :branch :source)= plists. (NB: that helper is workspaces/magit-coupled;
  =git worktree list= / =magit-list-worktrees= is the neutral alternative.)

** The path-archaeology problem (the rotten core)

Session identity + binding is reverse-engineered from the filesystem path
on every file open. User strongly dislikes this; it is fragile to any
reorg (workspaces is one of many).

- =jf/gptel--auto-init-session-buffer= on =find-file-hook= — runs on *every
  file open* (=commands.org:932=, comment at =:247=). Checks filename ==
  "session.org", then tries 3 hardcoded layout regexes (branch / nested-agent
  / flat-agent, =commands.org:270-309=), walks up with =../..= / =../../..=.
- Identity IS the directory basename: =jf/gptel--session-id-from-directory=
  = =(file-name-nondirectory dir)= (=filesystem.org:124-127=). Nothing stored;
  rename/move the dir -> identity silently changes.
- This hook does two jobs badly (binding + identity) and ignores a third
  (work root). Scope (Role 3) already avoids archaeology by being
  drawer-resident — the model A and B should copy.

* Emerging design principle (converged, not yet ratified)

# CORRECTION (2026-06-17 session): the "Preamble source in dir-locals" leg
# below is SUPERSEDED -- see [[*RESOLVED C]]. The preamble-source pointer goes
# in the DRAWER (written at create time from the integration-registry
# payload), not dir-locals. Also: the parenthetical "dir-locals can't set
# default-directory" (last bullet of this section) is FALSE -- verified by
# batch experiment that it can; see Thread C "Verified Emacs mechanics".

A single rule resolves the storage/activation questions and honors the
workspaces-decoupling constraint:

#+begin_example
  DRAWER (:PROPERTIES:)   per-ENTITY · varies · authoritative · co-located
    GPTEL_WORK_ROOT (cwd)      -> set default-directory from this in the mode hook
    GPTEL_SCOPE_* (already)    -> MUST agree with cwd, so keep them together
    [identity, if promoted off the path]
    written by whoever creates the session/agent (workspace OR plain cmd OR manual)

  DIR-LOCALS (.dir-locals.el)  per-TREE · shared · external-producer-writable
    jf/gptel--preamble-source  -> workspace points it at home.org; cascades down
    (optionally) mode activation for a whole tree
    inheritance is a FEATURE here (all sessions in a workspace share it)

  MAGIC-MODE-ALIST             activation only · content-addressed
    recognize the drawer signature (:GPTEL_PRESET: ...) -> gptel-chat-mode
    -> mode hook does all setup; RETIRES the global find-file-hook + path regexes
#+end_example

Principle in one line: *dir-locals where inheritance helps (shared tree
config), drawer where inheritance hurts (per-entity values that vary),
content-addressing for recognition (so filesystem layout stops mattering).*

Why each:
- *Work root in the drawer, not dir-locals:* dir-locals cascade downward, and
  agents live *under* the parent session dir — they would inherit the parent's
  cwd, but agents need a *different* cwd (a specific worktree). Per-entity
  values that must agree (cwd <-> scope) belong together in the drawer, which is
  already per-agent and authoritative for scope.
- *Preamble source in dir-locals:* the value is set by an external producer
  (workspace drops =.dir-locals.el= at =:home= pointing at =home.org=) and
  inherited by the whole tree. gptel reads buffer-local =jf/gptel--preamble-source=
  and never names a workspace symbol — textbook inversion of control,
  satisfying the decoupling constraint. Absent workspaces -> default preamble.
  (Mark vars =safe-local-variable= to avoid prompts.)
- *magic-mode-alist for recognition:* a session is recognized because it
  *contains* a gptel drawer, not by path or name. Same self-describing
  principle scope already uses, extended to activation. Once the mode activates
  by any means, the mode hook is the single setup point (and can set
  =default-directory= after find-file has run, sidestepping the "dir-locals
  can't set default-directory" edge).

** RESOLVED C — dir-locals seam subsumed by the integration-registry payload
   :PROPERTIES:

Original framing (now superseded): workspaces drops a =.dir-locals.el=
at =:home= setting =jf/gptel--preamble-source= (and other tree-shared
hints); the value cascades to every session under the tree; gptel reads
a buffer-local and never names a workspace symbol.

*Resolution: do NOT add a dir-locals layer. Write gptel's values into
the drawer at create time, sourced from the integration payload.*
Thread C largely collapses into [[*THREAD B]].

*** Why: two candidate seams, and one already exists

#+begin_example
  SEAM 1 — integration registry        (PUSH, at create time)   ALREADY EXISTS
    workspaces --(payload :home ...)--> gptel :on-create --> gptel writes drawer
    - workspaces names no gptel symbol     - gptel is the sole writer
    - value delivered exactly when gptel is creating the session

  SEAM 2 — .dir-locals.el at :home      (PULL, at open time)     what C proposed
    gptel :on-create --> writes .dir-locals.el --> gptel reads it on every open
    - gptel writes AND reads it (per "only gptel touches gptel vars")
#+end_example

Once we concede only gptel writes/reads its own variables, Seam 2's
writer and reader are *both gptel* -> dir-locals is not a decoupling
device, just a store gptel writes to itself. *The decoupling was always
coming from the registry payload, not from dir-locals.* Seam 2 is a
redundant second seam.

The only structural difference is push-at-create vs pull-at-open
(write-once-per-tree vs write-once-per-session). That pays off ONLY for
sessions gptel's create path never ran for:

  | session source           | covered by drawer (Seam 1)?      |
  |--------------------------+----------------------------------|
  | :on-create / :menu       | yes                              |
  | agent (--task)           | yes (needs own value anyway)     |
  | branch                   | yes (inherits parent drawer)     |
  | hand-authored / external | NO -- only dir-locals catches it |

Every row but the last funnels through gptel code already holding
=:home=. So the dir-locals-only benefit reduces to "a tree default for
sessions gptel never created" -- essentially the hand-authored case.

Cost of adding Seam 2: a git-committed artifact; merge-with-the-user's-
existing-=.dir-locals.el=; cascade noise into agent subdirs; and a new
*silent* fallback (an agent whose drawer write fails inherits =:home=
instead of its worktree -- reviving the cwd/scope-disagreement bug class
without a symptom).

*** Verified Emacs mechanics (batch experiments, 2026-06-17)

These overturn two claims made elsewhere in this doc (correction note
under [[*Emerging design principle]]):

- dir-locals *can* set =default-directory= (the doc's "can't" is FALSE).
- =default-directory= is =risky? nil=, =safe? stringp= in stock Emacs:
  settable, no prompt.
- Ordering is fixed: =gptel-chat-mode-hook= runs BEFORE file/dir-locals
  are applied (sees the file's own dir); =hack-local-variables-hook= runs
  AFTER (sees forced values). The codebase already relies on this split
  (=menu.org:895=, preset case).
- A drawer-driven override in =hack-local-variables-hook= BEATS a
  dir-locals value (a deep-subdir "agent" buffer overrode the inherited
  tree value). So dir-locals + drawer is a PRECEDENCE STACK, not XOR --
  this dissolves the doc's original objection (agents "would inherit the
  parent's cwd") but does NOT make the layer worth adding.

*** What dir-locals uniquely does (and why none of it serves a gptel goal)

A drawer only exists in files gptel authored and only means anything
once gptel's mode+hook are live. dir-locals is applied by Emacs core to
ANY file under the tree. That gap is real; the full inventory:

  | # | dir-locals-only capability                          | serves a gptel goal?               |
  |---+-----------------------------------------------------+------------------------------------|
  | 1 | reaches non-drawer buffers (src, home.org, dired)   | NO -- gptel must not touch those   |
  | 2 | retroactive: edit file -> existing sessions on open | marginal (beta/no-migration)       |
  | 3 | computed/self-relocating values via eval (move-safe)| marginal (eval risky; abs-anchored)|
  | 4 | single source of truth for tree (vs N drawers)      | low (values create-time-stable)    |
  | 5 | subtree override topology (.dir-locals.el in subdir)| NO -- drawer per-ENTITY is better  |
  | 6 | idiomatic/discoverable location                     | soft/UX only                       |

Every unique capability either serves a non-goal (1), is modelled
better by the drawer (5), or is marginal under beta/no-migration +

*** The reframe that keeps it clean

Capability #1 IS legitimately valuable -- "working anywhere in this
workspace sets my environment" -- but that is a *workspace-level* goal
about *non-gptel* buffers with *non-gptel* values (=default-directory=,
=compile-command=, project settings). gptel declining dir-locals does
NOT take that away: workspaces may drop its OWN =.dir-locals.el= with
non-gptel values anytime, naming no gptel symbol, fully decoupled.

#+begin_example
  gptel values (work-root, scope, preamble-pointer)  -> drawer via payload         no dir-locals
  workspace values (default-directory tree-wide ...)  -> workspaces' OWN .dir-locals  optional
#+end_example

*** Deferred escape hatch + re-entry condition

The precedence stack (drawer overrides an inherited dir-locals default)
is PROVEN to compose safely. Keep it in the back pocket: if hand-
authored sessions or a user-set workspace-wide default ever become worth
supporting, a dir-locals tree default can be added WITHOUT disturbing the
drawer-first design. It is an extension point, not the architecture.

Re-entry condition (the push-vs-pull boundary): the registry payload is
a PUSH seam (fires only at create). The day gptel gains a value that must
reflect the *current* tree state at every open -- rather than the state
frozen at create -- a pull seam (dir-locals or discovery) re-earns its
place. Checked 2026-06-17: work-root, scope, and the preamble-source
pointer are all create-time-stable; preamble *content* is resolved at
build time from whatever file the (stable) pointer names. Nothing needs
pull today.

* Cross-cutting constraints & invariants

- *Decoupling (hard):* gptel's work-root/default-directory/preamble setup must
  work with workspaces absent. Workspaces is one *producer* (writes drawer
  values at =:on-create=, drops =.dir-locals.el=); gptel is a consumer of
  intrinsic per-session data + buffer-locals. This *eliminates* "consult the
  live workspaces registry at runtime" as a work-root source.
- *Self-describing sessions:* prefer mechanisms where context travels with the
  session (drawer content; content-addressed activation) over mechanisms that
  reverse-engineer it from the path.
- *cwd must agree with scope:* whatever sets =GPTEL_WORK_ROOT= must keep it
  consistent with =GPTEL_SCOPE_*= (relative paths resolve into the scoped tree).
- *Beta, no-migration default:* on-disk formats may break between cycles; a real
  migration story is reserved for beta (mirrors workspaces persistence policy).
