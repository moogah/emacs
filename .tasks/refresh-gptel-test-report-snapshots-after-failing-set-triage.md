## Why this exists

`delete-old-presets` (cycle-1781944619) step 5 called for refreshing the `test-report`
snapshots. On inspection that was deliberately NOT done, because:

- **No snapshot enumerates presets**, so the deletion (this change's job) changes none of
  them. The cycle's new specs live under `config/gptel/presets/` (which has no tracked
  snapshot) or in chat/tools suites whose snapshots are stale.
- The snapshots are **inconsistent**: `config/gptel/chat/test/menu/test-report.txt` reads
  "0 ran"; `config/gptel/test-report.txt` is a scoped subset (ERT 22 / Buttercup 1350, not
  the full gptel suite); several others were captured GREEN.
- The live full suite carries a **stable pre-existing failing-set of 5 buttercup + 9 ERT**
  (scope add-to-scope / parallel-callback / filesystem-fs-ops / bash-parser pattern-flow /
  corpus-integration-002 / persistence-deserializer). Mechanically regenerating the
  snapshots now would inject those failures into files that don't currently show them —
  churn that misrepresents long-standing flakiness as a fresh regression.

Externalised because making the snapshots trustworthy is cross-cutting (spans scope,
bash-parser, sessions, filesystem) and requires first OWNING the pre-existing failing-set,
which is outside `gptel-fragment-presets`.

## Implementation steps

1. Triage the 5 buttercup + 9 ERT pre-existing failures (several already have `.tasks/`
   items: `gptel-preexisting-async-scope-test-failures`, `scope-and-bash-tool-test-failures-followup`,
   `fix-pattern-flow-*` / `fix-corpus-integration-002-cat-pattern-source`,
   `run-bash-multibyte-json-bugs`). Fix or explicitly quarantine each so the suite has a
   known-clean (or known-and-annotated) state.
2. Decide the snapshot contract: which directories get a `test-report.txt`, generated by
   what command (`make test-report DIR=...` / `./bin/run-tests.sh -d ... --report`), at
   what scope (per-subsystem vs aggregate). Remove the dead "0 ran" snapshots or regenerate
   them properly.
3. Regenerate all kept snapshots from the known state in one pass; commit.

## Verification

- `git diff --stat config/gptel/**/test-report.txt` reflects an intentional, explained refresh.
- A subsequent `make test-report` produces no diff (snapshots are now in sync with reality).
