## Files to modify

- `config/gptel/chat/mode.org` (modify) — extend the override to cover the full drawer span, not just per-line values
- `config/gptel/chat/mode.el` (regenerated by re-tangle)
- `config/gptel/chat/test/display/drawer-fontification-spec.el` (modify) — add the cross-line-emphasis regression scenario the cycle-7 fix didn't cover

## Why

Live user-testing finding (2026-05-23) against the cycle-7 fix's deployed state. Confirmed in `/Users/jefffarr/emacs-activities/drawer2-2026-05-23/session/branches/main/session.org`:

```
:GPTEL_SCOPE_READ: /**
:GPTEL_SCOPE_DENY: **/.git/** **/runtime/** **/.env ...
```

- Opening `/` on the READ value never closes on its own line. Org's emphasis parser scans forward and pairs it with the first `/` in the DENY value (between `**` and `.git`).
- Everything between (the rest of line 7, the entire `:GPTEL_SCOPE_DENY:` key text on line 8, and the value up to the closing `/`) is rendered in italic face.
- `org-hide-emphasis-markers` hides both `/` delimiters — so the user sees the closing `/` (in `**/.git`) as missing, plus an italicized key.

Cycle-7's `fix-scope-drawer-value-emphasis` installed `gptel-chat--drawer-value-matcher` with a per-line anchor (`^:KEY: value$`) and an OVERRIDE facespec on match-group 1 (the value text only). The fix correctly handles single-line cases (`:GPTEL_SCOPE_READ: /path/to/dir/` renders with face `org-property-value`, no italic). It does NOT handle:

1. **Cross-line italic spans.** Org's emphasis parser is happy to pair `/` delimiters across drawer lines; the per-line matcher cannot see the cross-line capture.
2. **Key text caught by value-originated emphasis.** When org's emphasis span starts in line N's value and closes in line N+1's value, the line N+1 KEY text (`:GPTEL_SCOPE_DENY:`) is between the open and close — italicized by org, untouched by our per-line override.
3. **Hidden boundary characters.** `org-hide-emphasis-markers` (t in most user configs including this one) hides the `/` delimiters of any matched italic span. Our face OVERRIDE doesn't restore visibility.

The contract from `register/boundary/chat-mode-session-display` override-A — "a property-drawer *value* in a chat-mode buffer renders as plain data and never inherits org /emphasis/ font-lock" — is broken. The cycle-7 implementation is necessary-but-insufficient.

## Implementation approach

The drawer span should render as plain text end-to-end. Pick one of:

**(a) Block org's emphasis parsing inside drawer spans.** Add a font-lock keyword that matches the entire `:PROPERTIES:` ... `:END:` span and uses an OVERRIDE facespec that both sets `org-property-value` face AND sets the `invisible` text-property to `nil` for the span. This restores boundary visibility AND beats org's italic face. Cleanest because it makes the whole drawer "plain". Recommended.

**(b) Pre-empt cross-line spans by neutralising emphasis-trigger characters in drawer values at font-lock time.** Set `font-lock-extra-managed-props` to include `invisible`, then in the matcher set both `(face org-property-value)` and `(invisible nil)` on the matched region — but extend the matcher to cover the entire drawer span, not per-line.

**(c) Disable `org-emphasis-alist` for chat-mode buffers entirely.** Too broad — it would also kill `/emphasis/` in chat-turn prose, which the override-A counter-assertion explicitly preserves. Don't do this.

Option (a) is the most targeted: it leaves chat-turn prose alone and addresses the drawer span as the single semantic unit it is.

## Implementation steps

1. In `config/gptel/chat/mode.org`, replace (or augment) `gptel-chat--drawer-value-matcher` with a span-level matcher `gptel-chat--drawer-span-matcher` that:
   - Anchors on `^[ \t]*:PROPERTIES:[ \t]*$`
   - Finds the matching `^[ \t]*:END:[ \t]*$`
   - Returns a match covering the whole region (the drawer content including keys and values, optionally excluding the `:PROPERTIES:` and `:END:` lines themselves since they have their own faces).
2. Install the keyword with OVERRIDE on face AND on `invisible nil`. The font-lock facespec form is:
   ```elisp
   (gptel-chat--drawer-span-matcher
    (1 'org-property-value t)
    (1 '(invisible nil) t))
   ```
   Add `invisible` to `font-lock-extra-managed-props` so font-lock cleans up when buffer text changes.
3. Re-tangle `config/gptel/chat/mode.org`; verify `./bin/tangle-org.sh` passes.
4. In `config/gptel/chat/test/display/drawer-fontification-spec.el` add the regression scenario:
   - Buffer text reproducing the user's case (`/**` value on line N, `**/.git/**` value on line N+1).
   - Assert the line-N+1 key `:GPTEL_SCOPE_DENY:` does NOT carry `italic` face.
   - Assert the `/` boundary characters in `**/` are NOT invisible (assert no `invisible t` text property on their positions).
   - Keep the existing single-line scenarios — they must still pass.
5. Run the chat suite.

The cycle-7 reconciliation note `boundary-chat-mode-session-display.md` recorded the override-A implementation-pinning-vs-observable trade-off as advisory; this finding is the cost of the trade-off — the structural assertion passes but the observable contract is broken on multi-line cases.

Meta-discovery for cycle-8 PM digest: the cycle-7 review's advisory about narrowing the override-A scaffold to "the observable-face contract" was correct *and* the reason was deeper than the reviewer articulated. The implementation-pinning scaffold passed because it asserted what the implementation does, not what the contract requires; the observable-face assertion (had it been preserved) would have failed against this case AND the implementor would have been forced to design for the span-level contract from the start. Worth a forward-mode prior: when an implementor proposes deviating from an observable-contract scaffold to an implementation-pinning one, the reviewer should require a co-located observable-contract spec to survive the deviation.
