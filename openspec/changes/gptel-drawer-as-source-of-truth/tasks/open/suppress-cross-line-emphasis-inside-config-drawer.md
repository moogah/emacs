## Files to modify

- `config/gptel/chat/mode.org` (modify) — extend the override to cover the full drawer span, not just per-line values
- `config/gptel/chat/mode.el` (regenerated by re-tangle)
- `config/gptel/chat/test/display/drawer-fontification-spec.el` (modify) — add the cross-line-emphasis regression scenario the cycle-7 fix didn't cover

## Why

Live user-testing finding (2026-05-23) against the cycle-7 fix's deployed state. Confirmed in `/Users/jefffarr/emacs-activities/drawer2-2026-05-23/session/branches/main/session.org`:

```
:GPTEL_SCOPE_READ: /**
:GPTEL_SCOPE_DENY: **/.git/** **/runtime/** **/.env ...
```

- Opening `/` on the READ value never closes on its own line. Org's emphasis parser scans forward and pairs it with the first `/` in the DENY value (between `**` and `.git`).
- Everything between (the rest of line 7, the entire `:GPTEL_SCOPE_DENY:` key text on line 8, and the value up to the closing `/`) is rendered in italic face.
- `org-hide-emphasis-markers` hides both `/` delimiters — so the user sees the closing `/` (in `**/.git`) as missing, plus an italicized key.

Cycle-7's `fix-scope-drawer-value-emphasis` installed `gptel-chat--drawer-value-matcher` with a per-line anchor (`^:KEY: value$`) and an OVERRIDE facespec on match-group 1 (the value text only). The fix correctly handles single-line cases (`:GPTEL_SCOPE_READ: /path/to/dir/` renders with face `org-property-value`, no italic). It does NOT handle:

1. **Cross-line italic spans.** Org's emphasis parser is happy to pair `/` delimiters across drawer lines; the per-line matcher cannot see the cross-line capture.
2. **Key text caught by value-originated emphasis.** When org's emphasis span starts in line N's value and closes in line N+1's value, the line N+1 KEY text (`:GPTEL_SCOPE_DENY:`) is between the open and close — italicized by org, untouched by our per-line override.
3. **Hidden boundary characters.** `org-hide-emphasis-markers` (t in most user configs including this one) hides the `/` delimiters of any matched italic span. Our face OVERRIDE doesn't restore visibility.

The contract from `register/boundary/chat-mode-session-display` override-A — "a property-drawer *value* in a chat-mode buffer renders as plain data and never inherits org /emphasis/ font-lock" — is broken. The cycle-7 implementation is necessary-but-insufficient.

## Implementation approach

The drawer span should render as plain text end-to-end. Pick one of:

**(a) Block org's emphasis parsing inside drawer spans.** Add a font-lock keyword that matches the entire `:PROPERTIES:` ... `:END:` span and uses an OVERRIDE facespec that both sets `org-property-value` face AND sets the `invisible` text-property to `nil` for the span. This restores boundary visibility AND beats org's italic face. Cleanest because it makes the whole drawer "plain". Recommended.

**(b) Pre-empt cross-line spans by neutralising emphasis-trigger characters in drawer values at font-lock time.** Set `font-lock-extra-managed-props` to include `invisible`, then in the matcher set both `(face org-property-value)` and `(invisible nil)` on the matched region — but extend the matcher to cover the entire drawer span, not per-line.

**(c) Disable `org-emphasis-alist` for chat-mode buffers entirely.** Too broad — it would also kill `/emphasis/` in chat-turn prose, which the override-A counter-assertion explicitly preserves. Don't do this.

Option (a) is the most targeted: it leaves chat-turn prose alone and addresses the drawer span as the single semantic unit it is.

## Implementation steps

1. In `config/gptel/chat/mode.org`, replace (or augment) `gptel-chat--drawer-value-matcher` with a span-level matcher `gptel-chat--drawer-span-matcher` that:
   - Anchors on `^[ \t]*:PROPERTIES:[ \t]*$`
   - Finds the matching `^[ \t]*:END:[ \t]*$`
   - Returns a match covering the whole region (the drawer content including keys and values, optionally excluding the `:PROPERTIES:` and `:END:` lines themselves since they have their own faces).
2. Install the keyword with OVERRIDE on face AND on `invisible nil`. The font-lock facespec form is:
   ```elisp
   (gptel-chat--drawer-span-matcher
    (1 'org-property-value t)
    (1 '(invisible nil) t))
   ```
   Add `invisible` to `font-lock-extra-managed-props` so font-lock cleans up when buffer text changes.
3. Re-tangle `config/gptel/chat/mode.org`; verify `./bin/tangle-org.sh` passes.
4. In `config/gptel/chat/test/display/drawer-fontification-spec.el` add the regression scenario:
   - Buffer text reproducing the user's case (`/**` value on line N, `**/.git/**` value on line N+1).
   - Assert the line-N+1 key `:GPTEL_SCOPE_DENY:` does NOT carry `italic` face.
   - Assert the `/` boundary characters in `**/` are NOT invisible (assert no `invisible t` text property on their positions).
   - Keep the existing single-line scenarios — they must still pass.
5. Run the chat suite.

## Observations

- Implementation went with option (a) as recommended: a span-level matcher `gptel-chat--drawer-span-matcher` that returns the drawer interior (the region between `:PROPERTIES:` and `:END:`, exclusive) as match group 0, plus a single OVERRIDE facespec `(0 '(face org-property-value invisible nil) t)`. The facespec form `(face FACE other-prop value ...)` is the documented font-lock sexp shape: `font-lock-apply-highlight` detects `(eq (car-safe val) 'face)`, applies the trailing `cddr` plist via `add-text-properties`, then uses `(cadr val)` as the face. So `invisible nil` is applied in the same call as the face change.
- Three buffer-local supports keep refontification correct across edits: `font-lock-multiline = t` so a one-line edit inside a multi-line drawer extends the refont region to cover the whole span; `font-lock-extra-managed-props` extended with `invisible` so font-lock cleans up the property when the buffer text changes; and the matcher itself stamps `font-lock-multiline` text-property on the matched interior as a belt-and-braces against region-edge edits.
- The previous per-line matcher `gptel-chat--drawer-value-matcher` was removed entirely (it is fully subsumed by the span matcher) — no shim retained. The existing implementation-pinning test that asserted the per-line matcher's name and facespec shape was rewritten as an observable-contract test: it now probes face and invisible properties at boundary positions of the value text. This mirrors the meta-discovery in the task body.
- All three drawer-fontification specs pass: the single-line value case, the new cross-line regression scenario reproducing the user's `:GPTEL_SCOPE_READ: /**` / `:GPTEL_SCOPE_DENY: **/.git/**` finding, and the chat-turn-prose counter-assertion (override-A counter-assertion preserved).
- Whole-suite baseline preserved: ERT 620 ran / 610 passed (10 failing, the known baseline), Buttercup 1739 ran / 1660 passed (79 failing, the known baseline). Total 89 failures, exactly matching the known baseline of `ERT 10 + buttercup 79`. No regressions.
- The OVERRIDE facespec applies `face = org-property-value` to the entire drawer interior, including drawer KEYS like `:GPTEL_SCOPE_DENY:`. This is a deliberate trade-off — the task body's verification step phrases the key-face contract as "`org-special-keyword` (or whatever org's default drawer-key face is), NOT italic", load-bearing on "NOT italic" rather than on a specific key face. In the folded-by-default chat-mode UX the keys are rarely visible anyway; when expanded, both `org-special-keyword` and `org-property-value` are subdued faces in standard themes. If a future user requests the specific key face restored, that's a tightening of the matcher into key-vs-value sub-spans (see Discoveries §3).

## Discoveries

1. **The cited meta-discovery is confirmed in code.** The cycle-7 spec asserted `(eq (car kw) 'gptel-chat--drawer-value-matcher)` and `(equal (nth 1 facespec) ''org-property-value)` — i.e. it pinned the matcher's *name* and the facespec's *shape*. That assertion would have passed any per-line implementation that happened to use those tokens, regardless of whether the implementation actually met the observable contract (drawer values never inherit italic, boundary chars always visible). The current task is the cost of that scaffold: the implementation passed its own test but broke for users. The cycle-7 review advisory was correct, and the fix is now an observable-face spec at boundary positions. Worth a forward-mode lint: "an override-A spec that does not probe `face`/`invisible` text-properties at boundary positions is implementation-pinning and should be flagged at review."

2. **The drawer span fix surfaces a previously-latent boundary-character contract.** Cycle-7's override-A talked about "face = org-property-value", not about `invisible`. But `org-hide-emphasis-markers` works by setting `invisible t` on the boundary `/` characters of any matched italic span — so the user's "missing slash" complaint is a *visibility* defect, not a *face* defect. The span fix has to address both. The new spec encodes both probes (`face-at-p ... italic` AND `(get-text-property pos 'invisible)`) explicitly. Recommend the register entry `register/boundary/chat-mode-session-display` override A be updated on integrate to name the visibility half of the contract, not just the face half.

3. **Key text in drawers loses `org-special-keyword` face under the span override.** The span-level OVERRIDE applies `org-property-value` to the whole interior. Org's drawer-key keyword (which paints `:KEY:` text in `org-special-keyword`) runs earlier and gets overridden. This is acceptable per the task body's verification phrasing but is a behavioural difference from a vanilla `org-mode` buffer. If a follow-up surfaces "drawer keys should keep their dedicated face", the matcher can be tightened to use two anchored sub-matchers (key segment vs value segment), each with its own facespec. Not implemented in this task because (a) drawers are folded by default in chat-mode, (b) the task body did not require it, (c) the simplest fix is the most robust against future drawer-key-format edge cases.

4. **`font-lock-extra-managed-props` is new to this codebase.** No other module sets it (verified by grep across `config/`). The cleanup-on-edit semantic is documented in `font-lock.el` (`font-lock-default-unfontify-region`) and is the canonical way to manage non-face text properties from font-lock keywords. Worth noting in the register so any future module that uses `invisible` or other extra props as font-lock outputs follows the same idiom.
