## REMOVED Requirements

### Requirement: Session file formats

**Reason**: The `metadata.yml` subsection of this requirement is obsolete. All of its keys either move to the `session.org` `:PROPERTIES:` drawer (`preset`, `parent_session_id`), are derivable from the session path (`session_id`, `type`), or are redundant with filesystem mtime (`created`, `updated`). The user has confirmed no existing sessions require legacy support.

**Migration**: The replacement contract is specified in two places:
- `session.org` format — see `gptel/chat-mode` Requirement: Session file format and persistence. The drawer is now authoritative for `preset` and `parent_session_id`.
- `scope.yml` format — split out as a separate requirement below (unchanged behavior).
- `branch-metadata.yml` format — split out as a separate requirement below (unchanged behavior).
- `session.org` format — split out as a separate requirement below (now authoritative, not sidecar).

The old `session-metadata` module (`config/gptel/sessions/metadata.org`) and its helpers (`jf/gptel--read-session-metadata`, `jf/gptel--write-session-metadata`, `jf/gptel--metadata-file-path`, `jf/gptel--update-metadata-timestamp`) are removed from the codebase.

## ADDED Requirements

### Requirement: scope.yml format

`scope.yml` SHALL be plain YAML (no frontmatter) with scope configuration sections generated from the preset's scope profile at session creation.

**Implementation**: Generated by `config/gptel/scope-profiles.org`.

#### Scenario: Scope created from preset profile
- **WHEN** creating session with preset `executor` (has `:scope-profile "coding"`)
- **THEN** loads `config/gptel/scope-profiles/coding.yml`
- **AND** expands `${project_root}` variables
- **AND** writes to branch directory as `scope.yml`

### Requirement: branch-metadata.yml format

Non-main branches SHALL have a `branch-metadata.yml` file in their branch directory with keys `parent_branch`, `created`, and optionally `branch_point_position`. Main branches SHALL NOT have a `branch-metadata.yml`.

This file is distinct from the removed `metadata.yml` — it carries branching-specific information, not session-level configuration.

**Implementation**: `config/gptel/sessions/branching.org`.

#### Scenario: Branch creation writes branch-metadata.yml
- **WHEN** running `M-x jf/gptel-branch-session`
- **THEN** creates `branches/<timestamp>-<name>/branch-metadata.yml`
- **AND** the file contains `parent_branch: <parent-name>`
- **AND** `main/` branches never have this file

### Requirement: session.org as authoritative session file

`session.org` SHALL be the single authoritative file for session content and per-buffer configuration. It is in `gptel-chat-mode` block format (`#+begin_user` / `#+begin_assistant` / nested `#+begin_tool`), with a `:PROPERTIES:` drawer at point-min that carries `GPTEL_PRESET`, `GPTEL_PARENT_SESSION_ID`, and any delta overrides (see `gptel/chat-mode` Requirement: Configuration drawer save on buffer save).

Initial content for a fresh session SHALL be:
```
:PROPERTIES:
:GPTEL_PRESET: <name>
[:GPTEL_PARENT_SESSION_ID: <id>]
:END:
#+begin_user

#+end_user
```

The `[:GPTEL_PARENT_SESSION_ID:]` line is present only for agent sessions.

#### Scenario: Fresh branch session.org has drawer
- **WHEN** creating a session with preset `coding`
- **THEN** `session.org` starts with a `:PROPERTIES:` drawer containing `:GPTEL_PRESET: coding`
- **AND** followed by `#+begin_user\n\n#+end_user\n`
- **AND** no `metadata.yml` is written in the branch directory

#### Scenario: Fresh agent session.org carries parent session id
- **WHEN** creating an agent session under parent `p-abc-20260424000000` with preset `executor`
- **THEN** the agent `session.org` drawer contains `:GPTEL_PRESET: executor` AND `:GPTEL_PARENT_SESSION_ID: p-abc-20260424000000`

## MODIFIED Requirements

### Requirement: Directory structure initialization

The system SHALL create this hierarchy for each session:

```
<session-dir>/
├── branches/<branch-name>/
│   ├── session.org             # Authoritative — preset & parent in drawer
│   ├── scope.yml               # Scope configuration
│   └── branch-metadata.yml     # Only on non-main branches
└── current -> branches/<branch-name>
```

`metadata.yml` is NOT created.

**Implementation**: `config/gptel/sessions/filesystem.org`

#### Scenario: New session creation
- **WHEN** running `M-x jf/gptel-persistent-session`
- **THEN** creates `branches/main/` directory
- **AND** `current` symlink points to `branches/main`
- **AND** no `branch-metadata.yml` in main branch
- **AND** no `metadata.yml` is created

#### Scenario: Branch creation
- **WHEN** running `M-x jf/gptel-branch-session`
- **THEN** creates `branches/<timestamp>-<name>/` directory
- **AND** includes `branch-metadata.yml` with parent reference
- **AND** updates `current` symlink to new branch
- **AND** no `metadata.yml` is created or copied

#### Scenario: Agent session creation
- **WHEN** `PersistentAgent` tool creates sub-agent
- **THEN** creates `branches/<branch-name>/agents/<preset>-<timestamp>-<desc>/`
- **AND** agent directory has `session.org` (drawer carries `GPTEL_PARENT_SESSION_ID`) and `scope.yml`
- **AND** no `metadata.yml` is created in the agent directory

### Requirement: Auto-initialization enables `gptel-chat-mode`

The auto-init hook (`jf/gptel--auto-init-session-buffer`) SHALL detect session files by matching the path pattern `*/branches/<branch-name>/session.org` (or the nested agent shape `*/<session-id>/branches/<branch>/agents/<agent>/session.org` and the flat legacy agent shape `*/<session-id>/agents/<agent>/session.org`). On match, it SHALL:

1. Extract `session-id` and `branch-name` from the path (branch-name defaults to `"main"` for the flat legacy agent shape that has no `branches/` segment).
2. Ensure the major mode is `gptel-chat-mode` (switching if necessary). This fires `gptel-chat-mode-hook`, which reads the `:PROPERTIES:` drawer and:
   - Applies `GPTEL_PRESET` via `gptel--apply-preset` with a buffer-local setter.
   - Overlays any non-preset drawer deltas (tools, model, backend, system, temperature, max-tokens, num-messages-to-send).
   - Sets `jf/gptel--parent-session-id` from `GPTEL_PARENT_SESSION_ID` when present.
3. Set the remaining buffer-local session variables not covered by the drawer overlay: `jf/gptel--session-id`, `jf/gptel--session-dir`, `jf/gptel--branch-name`, `jf/gptel--branch-dir`.
4. Register the buffer in `jf/gptel--session-registry`.
5. Update the `current` symlink to point at this branch (suppressed for the flat legacy agent shape, which has no `branches/` directory).

The hook SHALL NOT read `metadata.yml` (it no longer exists), SHALL NOT enable `gptel-mode` (minor mode), SHALL NOT invoke `gptel--save-state`, and SHALL NOT invoke `gptel--restore-state`. The drawer is authoritative; auto-init does not re-apply any preset after mode activation.

**Implementation**: `config/gptel/sessions/commands.org` — `jf/gptel--auto-init-session-buffer` via `find-file-hook`.

#### Scenario: Session file detection
- **WHEN** file matches `*/branches/<branch-name>/session.org` pattern
- **THEN** auto-init recognizes as branch session
- **AND** extracts session-id and branch-name from path
- **AND** enables `gptel-chat-mode` as the major mode
- **AND** does not attempt to read `metadata.yml`

#### Scenario: Agent file detection (nested)
- **WHEN** file matches `*/<session-id>/branches/<branch>/agents/<agent>/session.org` pattern
- **THEN** auto-init recognizes as nested agent session
- **AND** extracts both session-id and branch-name from the path
- **AND** enables `gptel-chat-mode` as the major mode
- **AND** updates the `current` symlink
- **AND** `jf/gptel--parent-session-id` is populated from the drawer's `GPTEL_PARENT_SESSION_ID`

#### Scenario: Agent file detection (flat legacy)
- **WHEN** file matches `*/<session-id>/agents/<agent>/session.org` pattern (no `branches/` segment)
- **THEN** auto-init recognizes as flat legacy agent session
- **AND** extracts session-id from the path
- **AND** sets branch-name to `"main"` as the default
- **AND** enables `gptel-chat-mode` as the major mode
- **AND** suppresses the `jf/gptel--update-current-symlink` side-effect

#### Scenario: Preset applied from drawer on open
- **WHEN** a session.org is opened whose drawer contains `:GPTEL_PRESET: coding`
- **THEN** `gptel-chat-mode` activation applies the `coding` preset via `gptel--apply-preset` with a buffer-local setter
- **AND** `gptel-mode` minor mode is NOT enabled

#### Scenario: Drawer overrides survive reopen
- **WHEN** a session file is reopened whose drawer contains `:GPTEL_PRESET: coding` and `:GPTEL_TOOLS: tool-a tool-b tool-c`
- **AND** the `coding` preset resolves `:tools` to `(tool-a tool-b)`
- **THEN** after mode activation, buffer-local `gptel-tools` is `(tool-a tool-b tool-c)`
- **AND** buffer-local `gptel--preset` is `coding`

#### Scenario: Agent parent-session-id restored from drawer
- **WHEN** an agent `session.org` is opened whose drawer contains `:GPTEL_PARENT_SESSION_ID: p-abc-20260424000000`
- **THEN** `jf/gptel--parent-session-id` is `"p-abc-20260424000000"` buffer-locally

### Requirement: Session creation

The system SHALL provide an interactive command for creating persistent sessions. `jf/gptel-persistent-session` SHALL create new sessions with `session.org` as the session file, populated with a pre-configured `:PROPERTIES:` drawer carrying `GPTEL_PRESET` (and `GPTEL_PARENT_SESSION_ID` for agents), followed by the chat-mode initial content (`#+begin_user\n\n#+end_user\n`).

`metadata.yml` is NOT created.

**Implementation**: `config/gptel/sessions/commands.org` — `jf/gptel-persistent-session`, `jf/gptel--create-session-core`.

#### Scenario: Create session with default preset
- **WHEN** run `M-x jf/gptel-persistent-session`
- **THEN** prompted for name
- **AND** generates session-id with timestamp
- **AND** creates directory structure
- **AND** creates `branches/main/session.org` whose content begins with `:PROPERTIES:\n:GPTEL_PRESET: <default-preset>\n:END:\n`
- **AND** followed by `#+begin_user\n\n#+end_user\n`
- **AND** does NOT create `metadata.yml`
- **AND** opens session.org in `gptel-chat-mode` with the preset applied via the drawer

#### Scenario: Create session with selected preset
- **WHEN** run `C-u M-x jf/gptel-persistent-session`
- **THEN** prompted for preset selection
- **AND** creates session with selected preset in the drawer
- **AND** scope from preset's profile written to `scope.yml`

#### Scenario: Create agent session
- **WHEN** `PersistentAgent` creates an agent under parent session `p-abc-20260424000000` with preset `executor`
- **THEN** agent `session.org` drawer contains `:GPTEL_PRESET: executor` AND `:GPTEL_PARENT_SESSION_ID: p-abc-20260424000000`
- **AND** does NOT create `metadata.yml`
