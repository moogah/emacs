* State machine вҖ” in-flight guard

#+begin_example
 upstream FSM state (gptel--fsm-last):
   nil/INIT в”Җв”Җ idle в”Җв”Җв”Җв”җ
   WAIT     в”Җв”Җ busy в”Җв”Җв”җв”Ӯ
   TYPE     в”Җв”Җ busy в”Җв”Җв”Өв”Ӯ   gptel-chat--in-flight-p

   ERRS     в”Җв”Җ idle в”Җв”Җв”Өв”Ӯ

                       в”Ӯ   NOT wedge the buffer
                       в”Ӯ

                       в”Ӯ
   waiting       в—Җв”Җ WAIT  (UI only вҖ” display layer, future mode-line)

                             nav.el regenerate blocked

The four modified session modules depend on chat-mode (hence chat-mode
is listed *before* sessions in =jf/enabled-modules=), but the
dependency is one-way вҖ” chat-mode itself knows nothing about sessions.

* Key architectural takeaway

Chat-mode is a downstream consumer of =gptel-request= with three
integration points:

1. The *callback closure* вҖ” =stream.el= handles every per-chunk event
   shape upstream emits (text, reasoning, tool-call, tool-result,
   terminal =t=/=nil=/=abort=).
2. The *chained FSM handler alist* вҖ” =send.el= installs a handler for
   each state (WAIT/TYPE/TOOL/DONE/ERRS/ABRT) that runs BEFORE
   upstream's, sets a buffer-local lifecycle indicator, and hands off.
   Never calls =gptel--fsm-transition=.
3. The *preset applier* вҖ” =menu.el= calls =gptel--apply-preset= with a
   buffer-local setter so preset keys (=:backend=, =:model=, =:system=,
   =:tools=, вҖҰ) land as buffer-local values, not globals.

No upstream patches. Every sibling reference inside chat-mode is
call-time, so the seven modules are reshuffleable вҖ” the load order in
=chat.el= is a readability convention that mirrors the data-flow
diagram above.
